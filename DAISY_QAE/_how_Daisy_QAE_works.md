
Let me walk through a concrete end-to-end example using a realistic scenario: "Build a WahWah + FDN Reverb effect on Daisy Field."

The Scenario
You're building a guitar effect patch in DVPE. The patch graph has two DSP blocks: WahWah and FDNReverb. You want to generate working Daisy C++ code from it.

Step 1 — DVPE Reads the Patch Graph
DVPE sees the patch has these node types:

nodeTypes = ['wah_wah', 'fdn_reverb', 'audio_in', 'audio_out']
Step 2 — 

apiReferenceInjector.ts
 Grounding
Before passing anything to the LLM, CodeGenerator.ts calls:

typescript
import { buildApiReferenceHeaderFromNodeTypes } from './apiReferenceInjector';
const header = buildApiReferenceHeaderFromNodeTypes(
    ['wah_wah', 'fdn_reverb', 'audio_in', 'audio_out'],
    (nodeType) => blockRegistry.getClassName(nodeType)
    // 'wah_wah' → 'WahWah', 'fdn_reverb' → 'FDNReverb'
    // 'audio_in' / 'audio_out' → undefined (skipped — not DSP classes)
);
apiReferenceInjector calls 

lookupClass('WahWah')
 and 

lookupClass('FDNReverb')
 against 

daisyApiIndex.ts
, then produces this block:

cpp
// ============================================================
// DAISYSP API REFERENCE — auto-generated by DVPE
// Grounding reference: verified signatures from DaisySP source.
// Use ONLY methods listed here. Do NOT invent method names.
// ============================================================
//
// [FDNReverb]  — daisysp::FDNReverb  (effects/fdn_reverb.h)
//   void Init(float sample_rate)  // [init]
//   void SetDecay(float decay)  // 0.9 - 0.999  [control]
//   void SetMix(float mix)  // 0.0 - 1.0  [control]
//   void SetDamping(float damping)  // 0.0 - 0.99  [control]
//   void SetDelayScale(float scale)  // 0.1 - 4.0 (room size)  [control]
//   void SetReverbTime(float rt60)  // [control]
//   float Process(float in)  // [audio]
//   void ProcessStereo(float in_l, float in_r, float *out_l, float *out_r)  // [audio]
//   void Clear()  // [control]
//
// [WahWah]  — daisysp::WahWah  (effects/wahwah.h)
//   void Init(float sample_rate)  // [init]
//   void SetFreq(const float &freq)  // 200.0 - 2000.0 Hz  [control]
//     ^ ⚠ SetFreq (NOT SetFrequency — old DAFX name removed)
//   void SetRes(const float &res)  // 1.0 - 20.0  [control]
//     ^ ⚠ SetRes (NOT SetQ — old DAFX name removed)
//   void SetDepth(const float &depth)  // 0.0 - 1.0  [control]
//   float Process(const float &in)  // [audio]
//
// ============================================================
This gets prepended to the prompt the LLM will see. The LLM reads it before writing a single line of code.

Step 3 — LLM Generates output.cpp
The LLM sees the grounding block at the top. It writes:

cpp
// ... [grounding block above] ...
#include "daisy_field.h"
#include "daisysp.h"
#include "../../MyProjects/DAFX_2_Daisy_lib/src/effects/wahwah.h"
#include "../../MyProjects/DAFX_2_Daisy_lib/src/effects/fdn_reverb.h"
#include "../../foundation_examples/field_defaults.h"
using namespace daisy;
using namespace daisysp;
DaisyField hw;
WahWah    wah;
FDNReverb verb;
volatile float p_wah_freq  = 800.0f;
volatile float p_wah_res   = 5.0f;
volatile float p_reverb_rt = 2.0f;
void AudioCallback(AudioHandle::InputBuffer in,
                   AudioHandle::OutputBuffer out, size_t size) {
    for(size_t i = 0; i < size; i++) {
        float sig = in[0][i];
        sig = wah.Process(sig);          // ← uses SetFreq/SetRes (not SetCutoff!)
        sig = verb.Process(sig);
        out[0][i] = out[1][i] = sig;
    }
}
int main(void) {
    hw.Init();
    float sr = hw.AudioSampleRate();
    wah.Init(sr);
    wah.SetFreq(p_wah_freq);            // ← correct — was SetFrequency before normalization
    wah.SetRes(p_wah_res);              // ← correct — was SetQ before normalization
    wah.SetDepth(0.8f);
    verb.Init(sr);
    verb.SetReverbTime(p_reverb_rt);
    verb.SetMix(0.4f);
    hw.StartAdc();
    hw.StartAudio(AudioCallback);
    while(1) {
        hw.ProcessAllControls();        // ← correctly in main loop, not callback
        p_wah_freq  = 200.f + hw.knob[0].Process() * 1800.f;
        p_reverb_rt = 0.5f  + hw.knob[1].Process() * 9.5f;
        wah.SetFreq(p_wah_freq);
        verb.SetReverbTime(p_reverb_rt);
        System::Delay(16);
    }
}
Why this is correct: The LLM used 

SetFreq
/

SetRes
 (not SetCutoff/SetQ/

SetFrequency
) because the grounding block showed only the correct names. Without the grounding block, a stock LLM would almost certainly write wah.SetFrequency(...) and wah.SetQ(...) — which no longer exist in the headers.

What Would Have Failed Without the System
Without the grounding block, a typical LLM would write:

cpp
wah.SetFrequency(800.0f);   // ❌ method no longer exists
wah.SetQ(5.0f);              // ❌ renamed to SetRes
The linter would catch both:

ERROR [HALLUCINATED-API] line:31  SetFrequency does not exist — use SetFreq() instead
ERROR [HALLUCINATED-API] line:32  SetQ does not exist — use SetRes() instead
And 

DAISY_HALLUCINATION_REFERENCE.md
 explicitly lists both renames in the DAFX section for anyone prompting an AI manually.

The Three Safety Nets Working Together
WahWah patch request
        │
        ├── [1] daisyApiIndex.ts → apiReferenceInjector.ts
        │         Grounding block prepended → LLM writes SetFreq/SetRes ✅
        │
        ├── [2] DAISY_HALLUCINATION_REFERENCE.md
        │         Manual AI prompting → same correct names ✅
        │
        └── [3] validate_daisy_code.py Rule 9
                  Even if [1] and [2] were skipped → linter catches it ✅
Each layer independently prevents the same class of bugs, so the system is robust even if one layer is bypassed.
