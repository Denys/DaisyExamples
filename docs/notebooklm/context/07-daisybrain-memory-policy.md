# DaisyBrain Memory Policy

## Purpose

DaisyBrain is not meant to be a full mirror of the repository. It should act as:

- strategic memory
- architectural recall
- cross-project synthesis
- a decision aid when local docs alone are not enough

## What Belongs In DaisyBrain

- high-signal repo structure
- workspace roles
- strategic roadmaps
- active project status summaries
- durable workflow rules
- cross-project comparisons and tradeoffs

## What Should Stay Outside DaisyBrain

- raw source dumps
- generated build output
- every historical artifact
- low-signal duplicated repo text
- deep project-specific implementation details that already live in local docs

## Memory Layering Rule

Use a layered memory strategy:

1. local project docs for exact truth
2. repo root docs for durable workflow and recent active context
3. DaisyBrain for strategic synthesis and recall

## Refresh Strategy

Refresh DaisyBrain when:

- a foundational workspace changes role
- a roadmap is materially updated
- an active project moves buckets
- a new board family or library workspace becomes strategically important

Do not refresh DaisyBrain for every small implementation detail.

## Notebook Source Strategy

Prefer these source types:

- curated synthesis files
- local README/CHECKPOINT/CHANGELOG docs for foundational workspaces
- active roadmap docs
- standards and validation doctrine

Avoid turning the notebook into a source dump from every project subtree.

## Decision Rule For New Sources

Add a source if it helps future agents answer:

- what is strategically important now?
- what is stable vs volatile?
- what is canonical vs transitional?
- what should be reused vs treated as an experiment?

If a source does not improve those answers, it probably does not belong in DaisyBrain.
