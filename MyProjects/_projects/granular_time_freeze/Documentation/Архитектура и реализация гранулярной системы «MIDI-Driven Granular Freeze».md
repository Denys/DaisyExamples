# Архитектура и реализация гранулярной системы «MIDI-Driven Granular Freeze»

## 1. Концептуальный обзор режима «Texture Freeze»

В проектировании современных инструментов для манипуляции тембром разделение процессов записи и воспроизведения является стратегическим рубежом, превращающим процессор эффектов в полноценный гранулярный инструмент. Для систем реального времени это разделение позволяет реализовать нелинейный доступ к памяти, превращая временной поток в пластичный звуковой объект.

**Определение Texture Freeze:** Концепция «замораживания текстуры» (Texture Freeze) формулируется как процесс принудительной деактивации указателя записи (Write Pointer) при сохранении высокой активности нескольких независимых указателей чтения (Read Pointers) [Source 546, 564]. В этом состоянии кольцевой буфер перестает обновляться, фиксируя «снимок» (snapshot) звукового события, который затем подвергается многократному сканированию гранулярными головками.

**Теоретический контекст:** В отличие от стандартного лупинга (looping), оперирующего целыми фразами, Texture Freeze опирается на классификацию DAFX для «замораживания» спектральных фрагментов [Source 540]. Согласно концепции Э. Роудса о микрозвуке, мы переходим от «потоков» (streams) к «облакам» (clouds) [Source 301]. Здесь звук перестает быть линейным; он становится статистическим распределением коротких зерен (grains), где плотность и хаотичность заменяют привычную структуру ритма и мелодии.

Этот переход от линейной записи к стохастическому воспроизведению требует жесткой архитектуры управления памятью, что подводит нас к реализации движка на базе кольцевого буфера.

## 2. Архитектура ЦОС: Движок кольцевого буфера

На платформе Daisy Field (STM32H7) управление памятью в реальном времени требует осознанного распределения ресурсов. Учитывая, что внутренняя память DTCM RAM ограничена, гранулярные буферы объемом в несколько секунд обязаны располагаться в секции `DSY_SDRAM_BSS`, чтобы избежать переполнения стека и обеспечить стабильную работу системы с низкой задержкой.

**Логика буфера на C++:** Для максимальной эффективности в `AudioCallback` мы избегаем использования оператора модуля `%`, если размер буфера является степенью двойки. Архитектор предпочтет битовую маску `& (BUFFER_SIZE - 1)`, что значительно экономит такты процессора ARM Cortex-M7 [Source 14.5.1].

```cpp
// Размещение буфера в SDRAM для поддержки текстур большой длительности
float DSY_SDRAM_BSS buffer[BUFFER_SIZE]; // BUFFER_SIZE должен быть степенью 2 (напр. 2^18)

void ProcessAudio(AudioHandle::InputBuffer in, AudioHandle::OutputBuffer out, size_t size) {
    for (size_t i = 0; i < size; i++) {
        if (!isFrozen) { // Состояние записи (Live)
            buffer[writePointer] = in[0][i];
            writePointer = (writePointer + 1) & (BUFFER_SIZE - 1);
        }
        // Указатели чтения (Read Heads) инкрементируются всегда
        float output = 0.0f;
        for (int g = 0; g < NUM_GRAINS; g++) {
            output += grains[g].Process(buffer);
        }
        out[0][i] = out[1][i] = output;
    }
}
```

**Гранулярные головки чтения:** Система реализуется как многоотводная линия задержки (multi-tap delay), где каждая «головка» — это независимый инстанс гранулы [Source 27, 297]. Использование переменных линий задержки позволяет каждой грануле иметь собственную скорость считывания, что критически важно для создания фазового смещения и предотвращения частотной гребенчатой фильтрации.

**Стохастический джиттер:** Для деструктуризации роботизированных повторов применяется рандомизированное смещение (jitter) позиции считывания [Source 660, 302]. Математически это реализуется добавлением белого шума [Source 7.4.1], масштабированного коэффициентом интенсивности, к базовому индексу `indexD`. Это превращает циклическое воспроизведение в органическое «облако».

Динамическое управление этими параметрами невозможно без внешней синхронизации, что требует внедрения системы MIDI-модуляции.

## 3. Архитектура управления: MIDI-синхронизированный степ-секвенсор

Синхронизация тембральных параметров с временной сеткой проекта (Tempo Sync) позволяет гранулярной текстуре дышать в такт с композицией, превращая статический шум в ритмический элемент.

**Концепция секвенсора параметров:** Мы переходим от традиционного секвенирования нот (Pitch) к модуляции параметров движка: размера гранул, плотности (Density) и позиции сканирования в буфере [Source 252]. Это создает «ритмический тембр», характерный для современной электронной музыки.

**Интеграция MIDI Clock:** Для точной работы используется обработчик байта `0xF8` (Timing Clock) [Source 4]. Поскольку `0xF8` не несет информации об абсолютном времени, система должна аккумулировать тики (24 тика на четвертную ноту) и измерять интервал между ними для вычисления `ticksPerStep` относительно текущей частоты дискретизации.

**Логика класса StepSequence:** В отличие от упрощенных Arduino-систем [Source 16, 17], наш класс `StepSequence` должен обрабатывать команду MIDI Start (`0xFA`) для сброса фазы и квантования шагов.

```cpp
class StepSequence {
public:
    void HandleMIDI(uint8_t byte) {
        if (byte == 0xF8) { // Clock
            if (++tickCounter >= ticksPerStep) {
                currentStep = (currentStep + 1) & 15; // Цикл 16 шагов
                tickCounter = 0;
                TriggerNewParams();
            }
        } else if (byte == 0xFA) { // Start
            currentStep = 0;
            tickCounter = 0;
        }
    }
private:
    uint32_t tickCounter = 0;
    uint8_t currentStep = 0;
    float sequenceData[16][NUM_PARAMS]; 
};
```

Интеграция программной логики с физическим интерфейсом Daisy Field завершает формирование инструмента.

## 4. Интерфейс и маппинг параметров (Daisy Field)

Эргономика в реальном времени требует минимизации когнитивной нагрузки. Маппинг должен быть интуитивным, где один регулятор влияет на сложное поведение системы.

**Реализация Super Knob:** Макрос «Chaos» является ключевым инструментом экспрессии. Он одновременно увеличивает плотность гранул и глубину джиттера [Source 859]. Маппинг параметров «Chaos» использует выпуклые (convex) характеристические кривые [Source 6.4.1, 8.4.2], что позволяет делать тонкие настройки на малых значениях и достигать экстремальной деструкции в конце хода регулятора.

|   |   |   |
|---|---|---|
|Регулятор|Параметр|Характеристика влияния [Source 8.4.2]|
|**Knob 1**|**Freeze State**|Переключение Write Pointer (Live/Frozen)|
|**Knob 2**|**Window Shape**|Морфинг между Triangle и Hanning|
|**Knob 3 (Chaos)**|**Density + Jitter**|Экспоненциальное нарастание хаоса|
|**Knob 4**|**Scan Pos**|Линейное смещение окна чтения|

**Визуализация на OLED:** Используя класс `OledDisplay` из _libdaisy_, мы визуализируем «визуальную грамматику» DaisySP [Source 856]. Обновление дисплея выносится в основной цикл `Main`, чтобы не вносить джиттер в высокоприоритетное прерывание аудио (Audio Interrupt). Состояние «Frozen» отображается через фиксацию формы волны с динамическими маркерами Read Heads.

Проработанный интерфейс минимизирует ошибки, но техническая чистота звука зависит от оптимизации алгоритмов.

## 5. Технические вызовы и оптимизация

Проектирование систем реального времени — это всегда борьба с артефактами.

**Управление задержкой (Latency):** При использовании больших SDRAM-буферов критически важен доступ к памяти. Архитектор должен минимизировать ветвления (`if/else`) внутри `for`-цикла обработки аудио. Вместо проверки состояния «Freeze» на каждом семпле, целесообразно использовать два разных указателя на функции обработки, переключаемых при смене режима.

**Устранение щелчков и артефактов:** Любой разрыв фазы на границах гранул порождает щелчок. Решением является оконная функция (Windowing), рассматриваемая в терминах «Array Multiplication» (перемножение массивов) [Source 8.4]. Применение окна Hanning (приподнятый косинус) обеспечивает спектральную гладкость, тогда как Triangle (линейное затухание) дает более перкуссионный характер [Source 115, 685]. Каждая гранула умножается на значения окна в процессе амплитудного фейда (Amplitude Fade), что гарантирует отсутствие фазовых разрывов.

## 6. Техническое резюме и спецификации

### Блок-схема сигнала

```text
[ MIDI IN ] |--> [ Clock Sync ] |--> [ Step Sequencer ]
             |                    |       | (Modulation)
             |                    V       V
[ Audio In ] ==> [ Circular Buffer ] ===> [ Multi-Tap Read Heads ]
             |   (SDRAM Section)  |       | (Windowing/VCA)
             |                    |       V
[ Control  ] |--> [ Chaos Macro ] |--> [ Summing/Mixing ] ==> [ Audio Out ]
```

### Контрольный список реализации

1. **Memory Mapping:** Определение буфера в `DSY_SDRAM_BSS`. Размер — 2^{18} семплов (5.46 сек при 48 кГц).
2. **DSP Optimization:** Замена `%` на `& (BUFFER_SIZE - 1)`. Использование `DMA` для обмена аудиоданными.
3. **Windowing:** Реализация `Windowing` как операции перемножения массивов (Array Multiplication) для каждой активной гранулы [Source 8.4].
4. **MIDI Handler:** Настройка прерывания UART для обработки `0xF8` (Clock) и `0xFA` (Start) с накоплением тиков для BPM-синхронизации.
5. **Audio Callback:**
    - Обновление `WritePointer` (только в Live режиме).
    - Итерация по `NUM_GRAINS`, вычисление позиции с учетом `Jitter`.
    - Применение оконной функции Hanning/Triangle.
    - Суммирование выходного сигнала.
6. **UI Thread:** Неблокирующее чтение энкодеров и обновление OLED-дисплея через библиотеку `libdaisy`.
